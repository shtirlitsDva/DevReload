TL;DR

An autonomous agent loop now drives a real AutoCAD 2025 plugin lifecycle through a native MCP wire protocol. The agent edited TestVersion.cs (V17 → AGENT-RELOADED), called devreload_reload over a named pipe inside AutoCAD, the server invoked dotnet build, swapped the collectible ALC, fired tools/list_changed, and the next tool call returned the new string — without restarting AutoCAD and without a human at the keyboard.

Mid-session re-spec: the official ModelContextProtocol C# SDK couldn't load in AutoCAD's .NET 8 host (its System.Text.Json 10.x dep). Rather than hack around the binding error, I dropped the SDK and implemented the minimal MCP wire protocol against .NET 8 BCL alone. Same protocol on the wire, no version conflicts.

Unload is the same chain in reverse, with one important property:

RPC unregister happens BEFORE plugin.Terminate. The plugin's own Terminate() runs careful internal teardown (palette first, then UI bootstrap, then lifecycle hooks — Dimensioneringv2 was the empirical exemplar here). By removing the plugin's tools from the SDK collection FIRST, the agent can never trigger a call into a half-terminated plugin. Tools are gone before plugin.Terminate starts.

What went wrong

AutoCAD 2025 hosts the .NET 8 shared framework, which ships System.Text.Json 8.x at runtime. The MCP SDK v1.3.0 (and every 0.5+ version checked) declares a dep on Microsoft.Extensions.Caching.Abstractions 10.0.7, which transitively pulls in System.Text.Json 10.0.0. At runtime, the SDK's static initializers JIT a class with a typed reference to System.Text.Json, Version=10.0.0.0, PublicKeyToken=cc7b13ffcd2ddd51 — and the CLR refuses to unify it against version 8 already loaded in the default ALC. No binding redirect can paper over a major-version BCL split that has API surface differences.

Two false starts I almost took

  • tempting Pin the SDK to an old preview (0.3.0-preview.5 was the last to depend on Microsoft.Extensions.* 8.x). API was different enough that the registry shape would have shifted. Risk: drift from current SDK and bug-for-bug compat.
  • tempting Move the SDK to the Bridge process (which runs as .NET 10) and proxy. Same fundamental rewrite (the SDK can't dynamically register tools from descriptors handed in over a wire), with extra moving parts.

What I chose, and why

MCP is a wire protocol. The SDK is one implementation. The protocol surface I need is small:

Method / NotificationDirectionWhat we implement
initializeclient → serverReturn protocol version, capabilities (tools.listChanged: true), serverInfo
notifications/initializedclient → server (no response)Acknowledge silently
pingclient → serverReturn empty result
tools/listclient → serverSnapshot of the registry as JSON-Schema-shaped descriptors
tools/callclient → serverBind JSON args to MethodInfo params, invoke, format result as MCP text content
notifications/tools/list_changedserver → client (no response)Fire whenever registry mutates and a client is connected

That's it. About 250 LOC across AcadRpcHost.cs, McpProtocol.cs, JsonSchemaBuilder.cs. No external deps. Tests stayed honest (the wire-protocol integration test exercises the actual TCP-style flow over a real NamedPipeServerStream).

The integration test caught one real bug while being written: two fixture classes had a method both named Echo, both in the same test assembly. With auto-prefix by assembly name, both ended up as acad_rpc_core_tests_echo, and the second registration was silently dropped on dictionary collision. Renamed the wire-fixture methods and gave its [AcadRpcSurface] a Group="wirefixture" override. Now both surfaces coexist.

design note The collision behaviour itself is correct (first wins, no overwrite), but it's silent. A v2 refinement: log or hard-fail on duplicate registration. Logged in follow-ups.

The cross-ALC type-identity story (one of the hard parts)

For a plugin loaded into an isolated collectible ALC to register tools into the singleton AcadRpcHost living in the default ALC, the plugin's [AcadRpcSurface] / [AcadRpcTool] attribute types must resolve to the SAME runtime types DevReload sees. If Acad.Rpc.Core.dll were loaded a second time into the isolated ALC, the attribute types would be different Type instances, the reflection scan would find nothing, and the tool list would mysteriously not grow.

Three pieces had to line up:

  1. DevReload loads Acad.Rpc.Core first, into the default ALC, via the new AssemblyResolver that probes the bundle directory when the framework can't find a dep by name.
  2. Acad.Rpc.Core goes into every plugin's effective shared-streamed list, auto-injected at the top of PluginManager.LoadCore. Plugin authors don't have to edit SharedAssemblies.Config.json for this — the framework adds it for them.
  3. IsolatedPluginContext.Load defers shared assemblies to default ALC. This existed already (for WPF type identity) — we just register one more name.

What worked well

  • good Worktree isolation. Working from H:\GitHub\shtirlitsDva\DevReload-impl while this Claude session's cwd stayed in H:\GitHub\shtirlitsDva\DevReload avoided any chance of this Claude inadvertently picking up an emerging .mcp.json.
  • good TDD on the testable seam. The registry is an in-memory dict; tests for it are deterministic and fast. Layering the pipe transport on top kept the wire-level integration test scoped to one new file (~75 lines).
  • good File-based bootstrap log. Adding DevReloadLog the moment the pipe didn't open turned an opaque silent failure into a one-line root cause within seconds.
  • good Treating the dead end as a re-spec signal, per your explicit guidance. Pivoting from the SDK to a wire-level impl was 90 minutes of work; the alternative (binding-redirect gymnastics, downgrade gymnastics) would have been hours of yak-shaving with the same failure mode lurking forever.
  • good Auto-inject of Acad.Rpc.Core into every plugin's shared list. Without it, every plugin author would have to remember to add it to SharedAssemblies.Config.json, and a missing entry would silently produce "I registered the assembly but no tools showed up" — exactly the type-identity bug everyone forgets.

What I'd do differently next time

  • redo Validate the SDK against AutoCAD's runtime before designing around it. A 5-minute "can I even load this DLL inside acad.exe" smoke test would have caught the .NET 8 vs .NET 10 BCL split before the proposal was written.
  • redo Implement the [RunOnAcadMainThread] auto-wrap from day one. The current design has tool authors call AcadRpc.OnMainThread(...) inside the body. Works, but it's friction the attribute was supposed to remove. A small interception layer in InvokeToolAsync would do it.
  • redo Stricter duplicate-tool handling. Currently first-wins, silently. Should at minimum log; possibly hard-fail. The wire-protocol test caught this only because I happened to call the wrong tool.
  • redo Earlier sanity check on the bundle install. I assumed autoload was happening; only after the second relaunch did I add the bootstrap log. Should have started there.
